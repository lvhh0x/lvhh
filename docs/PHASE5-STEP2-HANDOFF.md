## 2. Phase 5 Step 2는 무엇인가

### 2.1 한 줄 요약
박스 조합 결과에 **제품 사이즈 정보를 끝까지 보존**하고,
**서로 다른 사이즈 제품의 잔여 인박스를 "깔끔하게 떨어질 때만" 합치는** 규칙으로 개선한다.

### 2.2 배경 (도메인)
리본/열전사 필름(TTR) 제품을 출고할 때:
1. 제품 수량 → **인박스**(60/95/145 세 종류)로 포장
2. 인박스들 → **아웃박스 / 택배박스 / 낱개**로 묶음
3. 파렛트 선택 시 → 아웃박스를 파렛트에 적재

Step 1(완료)에서는 인박스를 **종류별로만** 합산해서 박스에 담았다 → **어느 사이즈 제품인지 정보가 사라짐.**
Step 2에서는 **사이즈별로 추적**하고, 합칠지 말지를 사이즈 단위로 판정한다.

## 4. ★★★ 새로 합의된 박스 조합 알고리즘 (가장 중요) ★★★

> 이 절이 이 문서의 핵심이다. 이전 구현(Step 3)과 **다르다.** 새 세션은 이 규칙대로 `outerbox.ts`를 다시 짠다.

### 4.1 인박스 분해 규칙 (엔진① — 안 바뀜, 참고용)

각 인박스는 **최대 용량**이며 그 이하로 채워도 된다.
"큰 박스부터 꽉 채우고, 남는 수량은 그게 들어가는 **가장 작은 인박스 1개**"에 담는다.

**사이즈별 인박스 용량(최대):**

| 제품 사이즈 | 145인박스 | 95인박스 | 60인박스 |
|------------|-----------|----------|----------|
| 40×300     | 30개      | 20개     | 10개     |
| 60×300     | 20개      | 10개     | (없음)   |
| 110×300    | 10개      | (없음)   | (없음)   |

**"몇 개부터 몇 개까지 어느 인박스" 직관 (사용자 설명):**
- 40사이즈: 1~10 → 60인박스 / 11~20 → 95인박스 / 21~30 → 145인박스
- 60사이즈: 1~10 → 95인박스 / 11~20 → 145인박스
- 110사이즈: 1~10 → 145인박스 (11~20이면 145인박스 2개: 10개+나머지)

**예:**
- 110사이즈 11개 → 145인박스 2개 (10개 채운 것 + 1개 채운 것)
- 110사이즈 25개 → 145인박스 3개 (10+10+5)
- 60사이즈 20개 → 145인박스 1개 (20개 = 145인박스 최대 용량)

### 4.2 ★ 단위 계수 (single source of truth로 만들 것) ★

**아웃박스 1개 = 60단위.** 기본 관계:
- 아웃박스 1개 = 145인박스 **4개**
- 아웃박스 1개 = 95인박스 **6개**
- 아웃박스 1개 = 60인박스 **10개**

각 인박스가 차지하는 단위:

| 인박스 | 아웃박스 1개에 들어가는 수 | 1개당 단위 |
|--------|---------------------------|-----------|
| 145인박스 | 4개  | **15단위** (60÷4) |
| 95인박스  | 6개  | **10단위** (60÷6) |
| 60인박스  | 10개 | **6단위**  (60÷10) |

**택배박스 1개 = 12단위.** (145=5, 95=3, 60=2 — 기존 INNER_UNITS 참고)

> **★ 사용자 합의 사항 (이번 세션 결정):** 이 계수들을 **`lib/company/units.json`으로 분리**하기로 함.
> **이유:** Claude가 이 단위표를 여러 번 헷갈렸음 → 값이 한 곳에만 박제되어 있어야 헷갈리지 않고,
> 코드도 그 한 곳에서만 읽으면 됨. Phase 6에서 Supabase로 이관할 때도 그대로 매핑되어 깔끔.
> **권장 구조:** JSON에 값 + 규칙 설명을 박아두고, `data.ts`가 읽어서 타입 입혀 재노출
> (JSON 맨 위에 이번 합의 규칙을 설명 필드로 같이 박아둘 것 — 코드 읽을 때마다 강제로 보게).
> (현재는 `lib/company/data.ts`의 `INNER_UNITS` 상수에 들어있음 — 이걸 JSON으로 옮기는 작업 포함)

### 4.3 ★ 박스 조합 핵심 규칙 (합칠지/분리할지) ★

**원칙: "같은 사이즈 먼저, 그리고 합쳤을 때 남는 게 없이 딱 떨어질 때만 합친다."**

단계별:

**[1단계] 같은 사이즈끼리 풀 아웃박스 먼저 추출**
- 각 사이즈별로 145인박스 4개(=60단위)가 차는 만큼 **풀 아웃박스**를 빼낸다.
- 남은 인박스는 사이즈별 "잔여"로 둔다.

**[2단계] 잔여를 아웃박스에 채우되 "딱 떨어지는 조합" 우선**
- 잔여 인박스가 **많은 사이즈부터** 아웃박스에 깔기 시작.
- 빈 단위에 **다른 사이즈를 통째로 넣어서 정확히 60단위가 되면 합친다.**
- **그 사이즈를 넣었을 때 쪼개져서 일부가 남으면 → 그 사이즈는 넣지 않고 대기.**
- 즉, "합쳐서 깔끔하게 떨어지는 사이즈"를 골라 채운다.

**[3단계] 남은(대기) 사이즈들 개별 처리**
- 각 대기 사이즈의 잔여 인박스를 자기들끼리:
  - 택배박스 1개(12단위)에 다 들어가고 2개 이상 → **택배박스**
  - 1개만 → **낱개(loose)**
  - 택배 초과 → **아웃박스(부분)** 로 그리디 처리

### 4.4 검증 예시 (사용자가 직접 제시 — 반드시 통과해야 함)

**예시 A: 40×180, 110×50, 60×40**
- 풀: 40→아웃박스1(120개), 110→아웃박스1(40개), 60→풀 없음
- 잔여: 40=60개(145인박스 2개), 110=10개(145인박스 1개), 60=40개(145인박스 2개)
  - ※ 단위로: 40잔여=2×15=30단위, 110잔여=1×15=15단위, 60잔여=2×15=30단위
- 40(30단위) + 60(30단위) = 60단위 딱 → **아웃박스 1개** (40과 60 혼합)
- 110(15단위, 145인박스1개) 남음 → **낱개 1개**
- **최종: 아웃박스 3개 + 낱개 1개(145인박스)**

**예시 B: 40×210, 110×20, 60×20**
- 풀: 40→아웃박스1(120개), 나머지 풀 없음
- 잔여: 40=90개(145인박스 3개=45단위), 110=20개(145인박스 2개=30단위), 60=20개(**145인박스 1개**=15단위)
  - ★주의: 60사이즈 20개 = 145인박스 1개 (20개가 145인박스 최대 용량이므로)
- 가장 많은 40(45단위=145인박스 3칸)을 아웃박스에 → 15단위(145인박스 1칸) 빔
- 다음 많은 110(30단위=2칸): 넣으면 45+30=75>60 쪼개짐 → 대기
- 60(15단위=1칸): 45+15=60 딱! → 40과 합쳐 **아웃박스 1개**
- 110(145인박스 2개=30단위)은 자동으로 → **택배박스 1개** (택배 12단위 초과? 145×2=10단위 ≤12 → 택배 가능)
- **최종: 아웃박스 2개 + 택배박스 1개**

**예시 C (분리해야 하는 경우): 110×30, 40×60**
- 풀: 둘 다 풀 없음 (110 30개=145인박스3개, 40 60개=145인박스2개)
- 만약 합치면: 110의 30개 + 40의 30개 = 아웃박스 1개, 근데 40이 30개 남음 → **남는 게 생기니 합치지 말 것**
- **최종: 110→아웃박스 1개(부분, 145인박스3개=45단위), 40→택배박스 1개(145인박스2개=30단위? → 택배 145×2=10단위≤12 OK)**
  - ※ 이 부분 단위 재확인 필요(§5)

### 4.5 이전 구현과 무엇이 다른가
- **이전(Step 3) 구현:** 잔여 전체를 FFD로 한 번에 패킹 후 "모든 bin이 60이면 합침, 아니면 전부 사이즈별 분리". → 너무 이분법적.
- **새 규칙:** 사이즈별로 **"통째로 넣어 딱 떨어지면 합치고 아니면 그 사이즈만 빼는"** 그리디. 합침과 분리가 사이즈 단위로 선택적.

## 5. 아직 확정 안 된 지점 (새 세션에서 사용자에게 확인)

1. **2단계 그리디의 정확한 탐색 순서**
   - "많은 사이즈부터 깔고, 빈 자리에 딱 맞는 사이즈를 찾는다"인데:
     - 빈 자리에 맞는 사이즈가 **여러 개**일 때 무엇을 우선? (더 많이 남은 것? 더 큰 사이즈?)
     - 한 아웃박스에 **3개 이상 사이즈**가 섞일 수 있는가?
   - → 의사코드로 구체화 후 사용자 확정 필요.

2. **택배박스 단위 재확인**
   - 택배박스 12단위에서 145인박스가 5단위인지(기존 INNER_UNITS) 재확인.
   - 예시 C에서 40 잔여(145인박스 2개)가 택배에 들어가는지 단위 검산 필요.

3. **잔여가 택배박스도 초과하는 복합 케이스**
   - 한 사이즈 잔여가 택배(12단위)를 넘으면 아웃박스(부분)로 가는데,
     그 아웃박스를 다시 다른 사이즈와 합칠 수 있는가? (아마 아니오 — 풀에서 이미 뺐으므로)

4. **단위 계수 JSON 분리 범위**

## 6. 검증 환경 (테스트 방법)

> 사용자 지침: 중요한 로직은 **반드시 로컬 테스트 환경에서 에러 0 확인 후** 푸시.

### 6.1 셋업
```bash
git clone --depth=1 https://github.com/lvhh0x/lvhh.git
cd lvhh
npm install --silent
npm install --save-dev ts-node tsconfig-paths   # 검증 스크립트 실행용
```

### 6.2 타입체크 (매 단계 필수)
```bash
npx tsc --noEmit        # 에러 0 이어야 함
```

### 6.3 검증 스크립트 실행
```bash
npx ts-node \
  --project tsconfig.json \
  -r tsconfig-paths/register \
  --compiler-options '{"module":"commonjs"}' \
  scripts/test-step4.ts
```
- `scripts/test-step4.ts`에 케이스가 들어있음. **§4.4 예시 A/B/C를 케이스로 추가**해서 돌릴 것.

### 6.4 푸시 방법
- 신규 파일: `github:create_or_update_file` (content 직접 전달, SHA 불필요)
- 기존 파일 갱신: `github:get_file_contents`로 SHA 받은 뒤 `create_or_update_file`에 sha 전달, 또는 `push_files`
- ⚠️ `github:push_files`에 **빈 배열 `[]`을 넣으면 "Cannot convert undefined or null to object" 에러** — 반드시 실제 `{path, content}` 객체를 넣을 것
- 배치 4~6 파일씩

## 8. 작업 원칙 / 사용자 지침 (반드시 준수)

### 8.1 진행 원칙 (가장 중요)
- **"실행", "수정", "추가" 또는 그에 준하는 명령이 없으면 절대 혼자 진행하지 않는다.**
- 코딩/수정 전에 반드시 **현재 상태 · 진행 상황 · 앞으로의 계획 · 발견된 모든 문제를 먼저 보고**하고,
  사용자와 의논해서 결정한 후 진행한다.
- 문제나 모호한 게 생기면 **멈추고 먼저 사용자에게 물어본다.**
- (이유: Claude가 혼자 판단해 올바른 로직을 틀렸다고 착각하고 고치려 한 사례 발생)

### 8.2 코딩 지침
1. 코딩은 사용자의 명령이 떨어져야 진행.
2. 파일이 올라오면 구조·설계목적·의도를 정확히 분석해 반영.
3. 앱 구현 시 상세 구현계획문서를 만든 후 완료 표시하며 구현.
4. 버그/에러/검증은 **테스트 환경 구축 후 검증**, 검증 내용을 로그 마크다운으로 만들어 다운로드 제공.
5. **`any` / `unknown` 타입 쓰지 말 것.**
6. 지속적으로 typecheck 실행해 새 문제 안 만들기.
7. 모든 작업/단계 완료까지 멈추지 말 것 (단, 8.1 우선).
8. 사용자에게 물어볼 때는 **먼저 물어볼 내용을 상세히 알려준 후** 선택지를 제시.
9. 한 파일에 몰아넣지 말고 **모듈 분리(UI/엔진/데이터/...) + 메인 통합** 구조(워크트리).
10. **백엔드/프론트엔드 분리** 개발.
11. 영어 정보는 한글로 번역해서 알려줄 것.
