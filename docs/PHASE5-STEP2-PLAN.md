## 1. 문제 정의 (왜 고치는가)

### 1.1 현실의 출고 방식과 어긋남
현재 엔진②(박스 조합)는 **모든 제품의 인박스를 사이즈 무관하게 종류별로 합산**한 뒤, 큰 인박스부터 FFD로 아웃박스(60단위)에 채운다. 그 결과 **서로 다른 사이즈가 한 박스에 어중간하게 섞일 수 있다.**

현실에서는 사이즈가 다른 제품을 한 박스에 마구 섞지 않는다. 빈 공간이 **딱 떨어지게** 채워질 때만 같이 넣고, **어중간하게 남으면 사이즈별로 분리**한다.

### 1.2 구조적 근본 원인 (코드 진단)
```
simulate.ts:  mergeInnerBoxCounts(perProduct)   ← 여기서 사이즈 정보 소실
                     ↓
outerbox.ts:  packIntoBoxes(innerTotals)         ← InnerBoxCount[](kind+count)만 받음
                     ↓
PackedBox.contents: InnerBoxCount[]              ← 사이즈/미터 없음
                     ↓
CompanyResult/BoxSvg: 사이즈 표기 불가
```
→ **인박스를 합산하는 순간 "어느 사이즈 것인지"가 사라진다.** 그래서 사이즈별 처리도, 화면의 "사이즈/미터/수량" 표기도 현재 구조로는 불가능하다. **타입에 사이즈 정보를 실어야** 두 문제가 동시에 풀린다.

## 3. 제품 마스터 데이터 (무게 포함 — `lib/company/data.ts` 기존값)

| 제품 | 풀 아웃박스 충진량 `fullOuterQty` | 145인박스당 | 95인박스당 | 60인박스당 | 풀 아웃박스 무게 `fullOuterWeight` |
|------|-----------------------------------|-------------|------------|------------|------------------------------------|
| 40×300 | **120개** | 30 | 20 | 10 | **15.92 kg** |
| 60×300 | **80개** | 20 | 10 | — (불가) | **15.80 kg** |
| 110×300 | **40개** | 10 | — (불가) | — (불가) | **15.00 kg** |

검산: 40×300 = 30 × 145인박스 4개 = 120개 ✅ / 60×300 = 20 × 4 = 80개 ✅ / 110×300 = 10 × 4 = 40개 ✅

### 3.1 ⭐ 핵심 제약 — 60인박스는 40사이즈 전용
- **60인박스는 오직 40×300 제품만 사용**한다. 60·110 사이즈는 60인박스를 쓸 수 없다.
- 이미 `data.ts`에서 `innerCapacity`의 60 항목이 60·110 제품은 `null`로 막혀 있고, 엔진①은 `null`을 건너뛴다. → **데이터로 이미 구조적 보장됨.** 추가 방어 코드는 넣지 않는다(프로그램 길어짐 방지).

### 3.2 무게 미상 항목 (현 placeholder, Phase 6에서 보강)
- 60인박스 tare = 0, 택배박스 tare = 0 → 무게 일부 과소 표시. `weightIncomplete` 플래그로 안내 중. (이번 개선 범위 아님)

## 4. 확정 알고리즘 (새 박스 조합 로직)

### 4.1 전체 파이프라인
```
① 제품(사이즈)별 인박스 분해          ← 엔진① 그대로, 단 사이즈별로 분리 유지(합산 X)
② 사이즈별 "풀 아웃박스(60단위)" 추출   ← 사이즈 정보 보존, 섞지 않음
③ 사이즈별 잔여(<60)를 한 풀에 모음
④ 잔여 풀을 60단위 FFD 패킹 후 빈칸 검사:
    ├─ 모든 bin == 정확히 60 (빈칸 0)  → 합침 (섞인 아웃박스로 확정)
    └─ 한 bin이라도 < 60 (빈칸 있음)   → 합치지 않고, 사이즈별로 같은 사이즈끼리 묶어:
          · 그 사이즈 잔여 인박스 1개   → 낱개 단독 출고
          · 그 사이즈 잔여 인박스 2개+  → 택배박스(cap 12) 시도 → 초과면 아웃박스(cap 60)
                                          (cap 넘으면 그리디로 박스 추가 반복)
```

### 4.2 핵심 판정 규칙 (가장 중요)
> **"합침"의 조건은 "잔여 합이 60의 배수"가 아니라, "FFD 패킹 후 모든 bin이 정확히 60(빈칸 0)"이다.**
>
> 단순 배수 검사는 불충분하다. 반례: 40잔여(35) + 60잔여(35) + 110잔여(50) = 합 120(=60×2, 배수 만족)이지만 어떻게 묶어도 깔끔한 60 두 박스가 안 나온다(35+35=70 초과 등). → FFD 후 빈칸 검사가 정확하다.

### 4.3 잔여 분리 시 "같은 사이즈끼리 묶기"
- ④에서 합침이 실패하면, 잔여를 **같은 사이즈끼리** 모아 박스에 담는다. (같은 사이즈가 여러 박스로 흩어지지 않게)
- 사이즈마다 독립 처리: 인박스 1개면 낱개, 2개 이상이면 택배(안 되면 아웃박스).

### 4.4 낱개 vs 택배 vs 아웃박스 경계
- **인박스 1개만 남음** → 낱개(단독 인박스로 출고)
- **인박스 2개 이상** → 무조건 박스에 담는다: 택배박스(12단위) 우선 → 초과 시 아웃박스(60단위)

## 5. 확정 테스트 케이스

### 5.1 신규 핵심 케이스 (질문 1 확정값)

**㉮ `40×300 ×90 + 60×300 ×100` → 아웃박스 2개 (합침)**
- 40: 90÷30 = 145인박스 3개 = 45단위 (부분)
- 60: 100÷20 = 145인박스 5개 = 75단위 = 풀 아웃박스 1개(60) + 부분 15
- 잔여 합 45+15 = 60 → **FFD bin 1개 정확히 60 → 합침**
- **결과: 아웃박스 2개** (풀 1 + 섞인 1)

**㉯ `40×300 ×90 + 60×300 ×120` → 분리**
- 40: 90÷30 = 145인박스 3개 = 45단위 (부분)
- 60: 120÷20 = 145인박스 6개 = 90단위 = 풀 아웃박스 1개(60) + 부분 30
- 잔여 합 45+30 = 75 → **FFD에 빈칸 발생 → 합치지 않음**
- **결과:**
  - 40 → 아웃박스 1개 (부분, 145인박스 3개)
  - 60 → 아웃박스 1개(풀) + 택배박스 1개 (145인박스 2개 = 택배 10단위 ≤ 12)

### 5.2 기존 합의 케이스 (BOX-MERGE-DISCUSSION 3.3, A~E)

| # | 입력 | 잔여 합 | 결과 |
|---|------|---------|------|
| A | 40×300 ×30 + 60×300 ×60 | 40:145×1(15) / 60:145×3(45) = 60 딱 | 아웃박스 1개 (섞음) |
| B | 40×300 ×60 + 60×300 ×40 | 40:145×2(30) / 60:145×2(30) = 60 딱 | 아웃박스 1개 (섞음) |
| C | 40×300 ×90 + 60×300 ×30 | 40:145×3(45) / 60:145×1+95×1 = 안 떨어짐 | 40:아웃박스1(부분) / 60:택배1(택배단위 5+3=8≤12) |
| D | 40×300 ×210 + 60×300 ×100 | 합 180 = 60×3 딱 | 아웃박스 3개 |
| E | 40×300 ×180 + 60×300 ×40 | 합 120 = 60×2 딱 | 아웃박스 2개 |

> ⚠️ A·B·D·E는 "잔여 합이 60 배수"이면서 동시에 "FFD 빈칸 0"이 성립하는지 구현 후 반드시 재확인한다(4.2 규칙).

### 5.3 기존 10개 검증 케이스 (Phase 5 Step 1)
- 기존 검증 스크립트(`verify.ts` 등) 위치를 먼저 확인한다.
- **새 규칙으로 10개를 전부 재계산**하고, **기대값이 바뀌는 케이스만 추려 사용자에게 먼저 보고**한다. (특히 복합 입력 케이스 — 110×50 + 60×30 등)
- 사용자 확정 후에만 기대값을 갱신한다.

## 6. 타입 변경 (`types/company.ts`)

사이즈 정보를 박스까지 실어 나르기 위해 다음을 추가/변경한다.

```ts
// 신규: 사이즈가 붙은 인박스 카운트
export interface SizedInnerCount {
  size: number;          // 40, 60, 110
  meter: number;         // 300
  kind: InnerBoxKind;    // 145 | 95 | 60
  count: number;         // 인박스 개수
  productQty: number;    // 이 인박스들에 담긴 제품 개수 (= count × innerCapacity[kind])
}

// 변경: PackedBox.contents 를 사이즈 포함 타입으로
export interface PackedBox {
  kind: OuterBoxKind | 'loose';
  contents: SizedInnerCount[];   // ← InnerBoxCount[] 에서 교체 (사이즈 3~4종 가능)
  filled: boolean;
  weight: number;
}
```
- 기존 `InnerBoxCount`는 `innerTotals`(전체 인박스 종류별 합계) 표시에 계속 쓸 수 있다.
- 화면 "사이즈/미터/수량" 표기는 `SizedInnerCount`에서 직접 뽑는다.

## 8. 파렛트 빈자리 채우기 (엔진③ 개선)

### 8.1 현재 동작
- 아웃박스만 층층이 쌓고, 택배박스·낱개는 시각적으로 맨 위에 얹기만 함.

### 8.2 새 동작 (사용자 확정)
> **마지막 층에 빈자리가 있으면 거기에 택배박스/낱개를 먼저 채운다. 마지막 층까지 꽉 차서 빈자리가 없으면 그때 맨 위에 얹는다.**

- 예: 700파렛트(층당 4개). 아웃박스 9개 → 1·2층 꽉(8개) + 3층 1개 → **3층에 빈자리 3칸** → 택배박스/낱개를 그 3칸에 채움.
- 마지막 층 빈칸 수 = `boxesPerLayer - lastLayerBoxes`.
- 빈칸이 택배/낱개 수보다 적으면, 채우고 남은 것은 맨 위에 얹음.
- 층당 박스 수는 파렛트별: 700=4, 900=6, 1100=9.

### 8.3 타입 영향
- `PalletStack`에 빈칸 채움 정보(예: `fillSlots`, `onTopExtras`)가 필요할 수 있음 → 구현 시 최소 필드만 추가하고 사용자 확인.
