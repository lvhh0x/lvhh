## 구현 순서 개요

쉬운 것(독립적·저위험)부터 → 엔진 대공사 → 가장 광범위한 것 순으로 배치했다.
각 Step은 **엔진(백엔드 계층) 먼저 → UI(프론트) 나중**, 그리고 **모듈 분리** 원칙을 따른다.

```
Step 1 — 박스 그리드 5열            (프론트 단독, 저위험)      ← 요구 1
Step 2 — 규격·오버행 엔진 신설       (엔진 신규 모듈)          ← 요구 3+4 (묶음)
Step 3 — 파렛트 슬롯/빈칸/오버 재설계 (엔진 대수정 + 프론트)     ← 요구 2
Step 4 — 원단 타입 도입             (타입+입력+엔진+표시 관통)  ← 요구 5
Step 5 — 통합 검증 & 빌드           (전체 회귀·빌드)
```

> **요구 3·4를 Step 2로 묶은 이유**: 둘 다 "파렛트+박스 합산 규격 계산"이라는 동일 엔진이다. 700(딱맞음)과 900(오버행)은 같은 공식의 다른 입력일 뿐이라 한 모듈에서 처리하는 게 옳다.
> **요구 2를 Step 3로 뒤에 둔 이유**: 슬롯/높이 계산이 Step 2의 오버행 규격과 결과 화면을 공유하므로, 규격 엔진이 선 완성돼야 깔끔하다.

각 Step 진행 규칙: **설계 재확인 → 사용자 승인 → 구현 → `tsc --noEmit` 통과 → 테스트 → VERIFY-LOG 기록 → 체크박스 완료.**

## Step 1 — 박스 그리드 5열 (요구 1)

### 목표
`[1] 박스 결과`의 박스 그림 영역을 한 줄에 **5개**씩 배치. (현재 flex-wrap으로 4개 자동 줄바꿈)

### 현재 상태
- 파일: `components/company/CompanyResult.tsx`
- 박스 그림 컨테이너: `display: flex; flexWrap: 'wrap'; gap: '12px'; alignItems: 'flex-end'`
- 각 `BoxSvg` size=120 → 부모 폭에 4개 들어가 자동 줄바꿈됨.

### 설계
1. 컨테이너를 `display: grid; gridTemplateColumns: 'repeat(5, 1fr)'; gap: '12px'`로 변경.
2. `alignItems: 'flex-end'` → grid에선 `alignItems: 'end'`로 (기존 하단정렬 유지).
3. 5개가 한 줄에 들어가되 박스가 너무 좁아지지 않도록, 필요 시 `[1] 박스 결과` 카드 폭 또는 `BoxSvg` size 미세조정 검토(브라우저 확인 후).
4. 박스 개수가 5의 배수가 아니어도 grid가 자동으로 다음 줄로 넘김(정상).

### 영향 범위
- `CompanyResult.tsx` 박스 그림 `div` 스타일 **한 곳만**. (지침 3 — 외과적 수정)

### 성공 기준 (verify)
- [ ] 아웃박스 5개 이상일 때 한 줄에 정확히 5개 표시
- [ ] 6번째 박스부터 다음 줄로 넘어감
- [ ] 하단 정렬(기존 동작) 유지
- [ ] 박스 그림/텍스트 안 잘림
- [ ] `npx tsc --noEmit` 통과

### 열어둔 질문
- 5열로 하면 각 박스가 좁아질 수 있음 → 카드 폭을 넓힐지 vs 박스를 줄일지는 **브라우저 확인 후 결정**(사용자와).

## Step 2 — 규격·오버행 엔진 신설 (요구 3 + 4)

### 목표
"규격 = 파렛트 + 박스 합산 외곽 치수"를 계산하는 **신규 엔진 모듈** `overhang.ts`를 만들고,
무게에 파렛트 자체 무게를 포함해 결과에 정확히 표시.

### 현재 상태
- 규격 표시: `CompanyResult.tsx`에서 `palletSpec.w × palletSpec.d` → **파렛트 원본 치수만** (박스 오버행 미반영)
- 높이: `pallet.ts`에서 `pallet.h + 315 × layers` (아웃박스만 기준)
- 무게: `simulate.ts`에서 `totalWeight`에 `palletTare` 이미 포함 O. 단, `[2]` "적재 무게"에 넘기는 `stackWeight`는 **아웃박스 무게만** → 택배/낱개 제품무게 누락.

### 설계

**(A) 신규 모듈 `lib/company/overhang.ts`**

박스 1종(아웃박스 355×315)이 파렛트에 어떻게 배열되는지로 합산 외곽을 계산.

```
입력: 파렛트 규격(w,d), 박스 규격(w,d), boxesPerLayer
출력: { footprintW, footprintD, overhangW, overhangD }
```

- 배열 규칙: 파렛트의 각 방향으로 박스를 몇 개 놓는지 결정.
  - 700(710×750, 층당4): 박스 355×315 → 2×2 배열 → 710×630 → 파렛트 안에 들어감(오버행 0)
  - 900(900×710, 층당6): 박스 세로(315)를 가로900 방향 3개=945, 박스 가로(355)를 세로710 방향 2개=710 → **945×710 (가로 45 오버행)**
- **⚠️ 확정 필요**: 700의 실제 배열이 2×2가 맞는지(층당4), 각 파렛트별 "가로 몇 개 × 세로 몇 개" 배열표를 확정해야 함. → **Step 2 착수 시 사용자 확인.**
- footprint = max(파렛트치수, 박스배열총폭). 오버행 = max(0, 박스배열총폭 − 파렛트치수).

**(B) `weight.ts` / `simulate.ts`**
- `[2] 파렛트 적재`의 "적재 무게"에 택배/낱개 무게도 포함하도록 `stackWeight` 계산 보정.
- 총무게 = 제품 + 인박스tare + 아웃박스tare + **파렛트tare** (이미 O, 유지).

**(C) `CompanyResult.tsx`**
- 규격 표시를 `footprintW × footprintD · 높이 H`로 교체.
- 오버행이 있으면 표시(예: "945×710 (박스 45mm 오버행)"). ← 추천 R3와 연계 가능.

### 영향 범위
- 신규: `lib/company/overhang.ts`
- 수정: `simulate.ts`(오버행 계산 호출·stackWeight 보정), `weight.ts`(적재무게), `CompanyResult.tsx`(규격 표시), `types/company.ts`(PalletStack에 footprint/overhang 필드 추가)

### 성공 기준 (verify)
- [ ] 700파렛트: 규격이 파렛트+박스 합산으로 표시 (오버행 계산 정확)
- [ ] 900파렛트: 규격 945×710 및 오버행 45mm 표시
- [ ] 적재 무게에 파렛트 자체 무게 포함
- [ ] 적재 무게에 택배/낱개 제품무게 포함
- [ ] 높이 = 파렛트높이 + 아웃박스높이 × 층수 (Step 3의 층수와 연동)
- [ ] `npx tsc --noEmit` 통과

### 열어둔 질문

- 오버행 시 무게는 변화 없음(맞음). 규격만 변함.

## Step 3 — 파렛트 슬롯/빈칸/오버 재설계 (요구 2)

### 목표
파렛트를 **항상 1개**로 고정하고, 슬롯 환산으로 적재 가능 여부·층수·높이를 정확히 계산.
빈칸에 택배/낱개를 배치하고, 슬롯 초과 시 "적재 초과 — 선택 불가" 표시.

### 현재 상태
- `pallet.ts`: `totalPallets = ceil(N / (bpl×5))` **다중 파렛트** 계산. 택배/낱개 미반영. 높이=아웃박스만.
- `CompanyPalletSvg.tsx`: 택배/낱개를 파렛트 맨 위 **별도 줄(topper)**에 그림. 빈칸(dashed)은 표시만.

### 설계 (PLAN §3-3 규칙 구현)

**(A) `pallet.ts` 재작성**
- 다중 파렛트 로직 제거 → **파렛트 1개 전제**.
- 슬롯 환산: `필요슬롯 = outerCount + courierCount + ceil(looseCount / 2)`
- 상한: `maxSlots = boxesPerLayer × 5`
- `필요슬롯 > maxSlots` → **적재 오버** 반환 (선택 불가 신호).
- 층수 = `ceil(필요슬롯 / boxesPerLayer)`
- 배치 순서: 아웃박스 채움 → 남은 빈칸에 택배박스(1칸씩) → 낱개(2개=1칸).
- 마지막 층 구성(어느 칸이 아웃/택배/낱개인지) 정보 생성 → SVG가 그림.
- 높이 = `pallet.h + 315 × 층수`.

**(B) `simulate.ts`**
- `calcPallet`이 오버 신호를 주면 결과에 `palletOverflow: true` 담아 UI에 전달.

**(C) `CompanyPalletSvg.tsx` 수정**
- topper(별도 줄) 제거 → 택배/낱개를 **실제 빈칸 슬롯 위치**에 그림(택배=1칸, 낱개2개=1칸).
- 빈 슬롯이 진짜로 남으면 그때만 dashed 표시.

**(D) `CompanyResult.tsx`**
- `palletOverflow`면 `[2]` 대신 "적재 초과 — 이 파렛트는 선택할 수 없습니다" 경고 표시.
- (추천 R2 채택 시) 가능한 파렛트 대안 안내.

### 영향 범위
- 수정: `pallet.ts`(대수정), `simulate.ts`, `CompanyPalletSvg.tsx`(대수정), `CompanyResult.tsx`, `types/company.ts`(PalletStack 필드·overflow)

### 성공 기준 (verify)
- [ ] 700파렛트 아웃박스 21개↑ → 적재 오버 표시
- [ ] 900파렛트 아웃박스 31개↑ → 적재 오버 표시
- [ ] 아웃박스 18개 + 택배 2개 → 빈칸 2칸에 택배 배치, 층수/높이 정확
- [ ] 아웃박스 18개 + 택배 5개 → 필요슬롯 23 > 20 → 적재 오버
- [ ] 낱개 3개 → 2슬롯 차지 확인
- [ ] SVG에 택배/낱개가 빈칸 위치에 그려짐
- [ ] `npx tsc --noEmit` 통과

### 열어둔 질문
- 오버 시 대안 파렛트 추천(R2) 넣을지 → 사용자 결정.
- 낱개와 택배가 동시에 빈칸을 다툴 때 우선순위(택배 먼저 vs 낱개 먼저) → 기본 "택배 먼저" 제안, 착수 시 확인.

## Step 4 — 원단 타입 도입 (요구 5)

### 목표
입력에 **원단 타입**(자유입력 텍스트)을 추가하고, 같은 원단 우선 그룹핑으로 박스를 조합.
개수·무게 계산에는 영향 없음.

### 현재 상태
- 입력: `CompanyParams.tsx` — 사이즈/미터/수량 3필드.
- 타입: `ProductInput { size, meter, qty }`, `SizedInnerCount { size, meter, kind, count, productQty }`.
- 그룹핑: `outerbox.ts`가 `(size, meter)` 키로만 그룹핑.

### 설계

**(A) 타입 확장 `types/company.ts`**
- `ProductInput`에 `fabric: string` 추가.
- `SizedInnerCount`에 `fabric: string` 추가 (박스까지 원단 정보 전달).
- `FlatInner`/`Chunk`(outerbox 내부)에도 `fabric` 부착.

**(B) 입력폼 `CompanyParams.tsx`**
- 행 입력 순서: **원단 타입 → 사이즈 → 미터 → 수량**.
- 원단 타입: `<input type="text">` (자유입력). placeholder 예: `B220`.
- `RowState`에 `fabric` 추가, `handleRun` 필터/매핑에 반영.
- 원단 미입력 허용 여부 → 착수 시 확인(기본: 필수).

**(C) 인박스 분해 `innerbox.ts`**
- `decomposeToInnerBoxes`가 `product.size/meter`처럼 `fabric`도 `SizedInnerCount`에 실어 전달.
- (fabric은 계산에 안 쓰이고 통과만 — passthrough)

**(D) 박스 조합 `outerbox.ts` (핵심)**
- 그룹핑 키를 `(size, meter)` → **원단 우선**으로:
  1. 먼저 **같은 fabric 안에서** 풀 아웃박스 추출.
  2. 같은 fabric 잔여로 60이 안 되면 → **다른 fabric과 exact-fill 섞기 허용** (PLAN §3-2).
  3. 나머지 FFD 잔여 처리(기존 유지).
- `packIntoBoxes`의 1·2단계 그룹핑 로직을 fabric 계층 추가해 재구성. **개수 규칙(60/12)은 절대 불변.**

**(E) 표시 `BoxSvg.tsx` / `CompanyResult.tsx`**
- 박스 내용 텍스트에 원단 타입 표기(예: "B220 40×300 145×4"). ← 표시용, 선택적.

### 영향 범위
- 수정: `types/company.ts`, `CompanyParams.tsx`, `innerbox.ts`, `outerbox.ts`(핵심), `BoxSvg.tsx`, `CompanyResult.tsx`
- (master.json은 원단 규칙이 계산에 없으므로 **변경 불필요**. 단 Phase 6 DB화 시 원단 테이블 신설 고려 — PLAN에 메모됨)

### 성공 기준 (verify)

- [ ] `B220 40×300 60개` + `B324 60×300 20개` → 45단위 → 부분 아웃박스 1개
- [ ] `B220 60×300 40개` + `B325 110×300 20개` + `B324 40×300 60개` → 아웃박스 1개 + 택배박스 1개

## Step 5 — 통합 검증 & 빌드

### 목표
전체 시나리오 회귀 테스트 + 빌드 검증 + 문서 갱신.

### 설계
1. **테스트 스크립트 보강** (`scripts/`): 기존 18케이스 + 이번 신규 케이스(원단 그룹핑, 슬롯 초과, 오버행)를 추가. (추천 R1)
2. **`npx tsc --noEmit`** 최종 0 에러.
3. **`npm run build`** (샌드박스는 폰트 스텁 임시 적용, push 금지) → Compiled successfully.
4. **브라우저 확인**: `lvhh.vercel.app/company`에서 5개 요구사항 전부 눈으로 검증.
5. **문서 갱신**: `docs/PROGRESS.md`, `docs/MASTER.md`에 이번 개선 반영.
6. **VERIFY-LOG 완성**: 모든 검증 결과 기록 후 다운로드 제공.
