> **아래 두 tare는 현재 `0` placeholder다. 값이 확정되면 반드시 master.json에서 교체하고 `weightIncomplete` 플래그 로직을 재점검할 것.**

| 위치 | 항목 | 현재값 | 상태 |
|------|------|--------|------|
| `innerBoxes[kind=60].tare` | 60인박스 공무게 | `0.00` | ❌ 미상 — 사용자가 추후 제공 |

결과 화면에 "× 60인박스·택배박스 무게 미반영" 안내가 뜬다. **값 확정 시 이 안내 로직도 함께 손봐야 함.**

## 3. ⭐ 핵심 도메인 규칙 (이번 개선으로 확정된 것)

### 3-1. 단위(unit) 개념 — 절대 혼동 금지

> **단위는 "물건 크기"가 아니라 "그 박스를 몇 % 채웠나"의 점수다.**
> 같은 인박스라도 **어느 박스를 채우느냐**에 따라 점수가 다르다.

| 인박스 | 아웃박스 채울 때 | 택배박스 채울 때 |
|--------|------------------|------------------|
| 145 | 15단위 | 5단위 |
| 95 | 10단위 | 3단위 |
| 60 | 6단위 | 2단위 |

- 아웃박스 총용량 = **60단위** (예: 145×4=60)
- 택배박스 총용량 = **12단위** (예: 145×2=10, 95×4=12)
- 예) 145인박스 2개 = 아웃박스 기준 30단위(반쪽) = 택배박스 기준 10단위(거의 참) → **택배박스로 포장**

### 3-2. 박스 조합 규칙 (기존 유지 + 원단 우선순위 추가)

1. 같은 원단끼리 먼저 60단위 풀 아웃박스 추출
2. 같은 원단으로 60이 안 채워지면 → **다른 원단과 섞어서라도** 정확히 60 채움 (exact-fill)
3. 남은 잔여는 FFD로 묶어 판정:
   - 60단위 정확 → 풀 아웃박스(filled=true)
   - 택배단위 ≤ 12 & 인박스 2개↑ → 택배박스 / 1개 → 낱개
   - 택배단위 > 12 → 부분 아웃박스(filled=false)

### 3-3. 파렛트 적재 규칙 (2번으로 확정)

- **파렛트는 항상 1개.** (다중 파렛트 없음)
- 모든 파렛트는 아웃박스 기준 **최대 5층**.
- **슬롯 환산**:
  - 아웃박스 1개 = **1슬롯**
  - 택배박스 1개 = **1슬롯**
  - 낱개(인박스) 2개 = **1슬롯** → 낱개 홀수는 `ceil(낱개/2)` 슬롯 (반칸 버림)
- **필요 총 슬롯 = 아웃박스수 + 택배박스수 + ceil(낱개수 / 2)**
- **파렛트 상한 슬롯 = boxesPerLayer × 5**
  - 700파렛트: 4×5 = **20슬롯**
  - 900파렛트: 6×5 = **30슬롯**
  - 1100/플라스틱: 9×5 = **45슬롯**
- 필요 총 슬롯 > 상한 슬롯 → **"적재 초과 — 이 파렛트는 선택할 수 없습니다"** 표시
- 층수 = `ceil(필요 총 슬롯 / boxesPerLayer)`
- 배치 순서: 아웃박스 먼저 → 남는 빈칸에 택배박스 → 그다음 낱개

### 3-4. 규격·오버행 규칙 (3·4번으로 확정)

- **규격 = 파렛트에 박스를 올렸을 때 파렛트+박스를 합한 외곽 규격** (파렛트 단독 규격이 아님).
- 파렛트 가로/세로 방향으로 박스를 배열했을 때, 박스 배열 총폭이 파렛트를 넘으면 그 **오버행(초과분)**까지 규격에 반영.
- 예) 900파렛트(900×710) + 아웃박스(355×315):
  - 가로 900 방향 ← 박스 세로(315) 3개 = 945 → **가로 945** (45 오버행)
  - 세로 710 방향 ← 박스 가로(355) 2개 = 710 → **세로 710** (딱 맞음)
  - → 합산 규격 **945 × 710**, 높이 = 파렛트높이 + 아웃박스높이 × 층수
- 배열 방식(가로/세로 몇 개)은 파렛트별 `boxesPerLayer`와 박스 치수로 계산 (Step에서 상세 설계).

### 3-5. 원단 타입 규칙 (5번으로 확정)

- 입력 순서: **원단 타입 → 사이즈 → 미터 → 수량** (원단 타입이 맨 앞).
- 입력 방식: **자유입력 텍스트** (드롭다운 아님. 종류가 앞으로 많아짐).
- 예: `B220` = 종이 인쇄용, `B324` = 옷 라벨용, `B325` = 비닐/유리용 (예시일 뿐, 종류 무한).
- 규칙:
  - 원단 타입은 **인박스/아웃박스 개수 계산에 영향 없음** (60/12 단위 규칙 그대로).
  - **무게에도 영향 없음.**
  - 오직 **그룹핑 우선순위**에만 영향: 같은 원단끼리 먼저 묶고, 안 채워지면 섞어서 60 채움.
- 예) `B220 110×300 30개(45단위)` + `B325 110×300 30개(45단위)` → 원단 달라 각각 → **아웃박스 2개** (안 섞음).

## 4. 아키텍처 방침 (지침 9·10 반영)

### 4-1. 모듈 분리 (워크트리 방식)

기존 구조가 이미 모듈 분리돼 있어 이를 유지·확장한다. 한 파일에 몰아넣지 않는다.

```
lib/company/               ← 엔진 (계산 담당, 프론트와 분리)
  ├─ master.json           · 데이터 단일 출처
  ├─ data.ts               · JSON↔타입 매핑
  ├─ innerbox.ts           · 엔진① 인박스 분해
  ├─ outerbox.ts           · 엔진② 박스 조합 (← 5번 원단 그룹핑 추가)
  ├─ pallet.ts             · 엔진③ 파렛트 적재 (← 2·3·4번 대폭 수정)
  ├─ weight.ts             · 엔진④ 무게 계산 (← 3번 파렛트 무게)
  ├─ overhang.ts (신규)    · 엔진⑤ 규격·오버행 계산 (3·4번 전용, 신규 모듈)
  └─ simulate.ts           · 통합 오케스트레이터

components/company/         ← UI (프론트, 표시 담당)
  ├─ CompanyParams.tsx     · 입력폼 (← 5번 원단 입력란)
  ├─ CompanyResult.tsx     · 결과 패널 (← 1번 그리드, 3·4번 규격 표시)
  ├─ CompanyPalletSvg.tsx  · 파렛트 그림 (← 2번 빈칸 배치)
  ├─ BoxSvg.tsx            · 박스 그림 (← 5번 원단 표시 옵션)
  └─ CompanyTile.tsx       · 타일

types/company.ts           ← 타입 정의 (← 원단 필드 추가)
```

> **신규 모듈 `overhang.ts`**: 규격·오버행 계산은 개념이 독립적이고 로직이 커서 별도 모듈로 분리(지침 9). pallet.ts에 섞으면 파일이 비대해짐.

### 4-2. 백엔드/프론트엔드 분리 (지침 10)

- 이 기능의 계산 로직(`lib/company/**`)은 **순수 TypeScript 엔진 = 프론트 내 백엔드 계층**. UI(`components/**`)와 명확히 분리.
- 현재 Company 시뮬레이션은 Python 서버를 쓰지 않고 Next.js 내부에서 계산(Stock 시뮬레이션만 Railway Python 사용). 이 구조 유지.
- 향후 Phase 6에서 데이터가 Supabase로 가면 `data.ts`가 그 경계 계층이 됨.

## 5. 작업 원칙 (항상 유지 — 사용자 지침)

1. **코딩 명령 없이 코딩 금지.** 명시적 "실행/수정/추가" 명령 후에만 진행.
2. **파일이 올라오면 구조·목적·의도·목표를 정확히 분석해 반영.**
3. **구현 전 반드시 상세 계획 문서 작성 → 완료 표시하며 구현.** (이 문서 세트가 그 역할)
4. **테스트 환경 구축 후 검증 → 모든 과정을 로그 마크다운으로 기록 → 검증 완료 후 다운로드 제공.**
5. **`any` / `unknown` 타입 금지.**
6. **지속적으로 `npx tsc --noEmit` 실행해 새 문제 없게 유지.**
7. **모든 작업·단계 완료까지 멈추지 않음.** (단, 각 단계 완료 시 사용자와 검토 후 다음 단계)
8. **사용자에게 물을 땐 먼저 물을 내용을 상세히 설명한 후 선택지 제시.**
9. **모듈 분리(워크트리): UI/엔진/데이터/카테고리 등 담당별로 파일 분리, 메인이 통합.**
10. **백엔드/프론트엔드 2분류로 개발.**
11. **영어 정보는 한글로 번역해 전달.**
